# Governance Policy v2

## Purpose
This file defines how memories are added, updated, deprecated, superseded, or overlaid.

## Mutation rules
- Add a new memory when a concept becomes durable, reusable, and likely to recur.
- Update an existing memory when the concept is materially the same but the implementation becomes more accurate.
- Supersede a memory when a newer object replaces the older object functionally.
- Deprecate a memory when it should not be used going forward.

## Supersession rules
Use supersession when:
- the old logic is still historically meaningful,
- but the new logic should be the default.

## Deprecation rules
Use deprecation when:
- the old logic is no longer safe,
- no longer aligned,
- or no longer economically or architecturally viable.

## Review cadence
- Review Tier 1 memories quarterly or on major architecture changes.
- Review client namespaces at onboarding and renewal.
