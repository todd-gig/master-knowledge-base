1\. Design Objectives
---------------------

1.  Every interaction (voice or text) creates **causal evidence**, not just logs.
2.  **Nodes** represent entities, states, and abstract concepts.
3.  **Edges** represent directional influence: `A → B` with strength, type, and evidence.
4.  **Importance**, **trust**, and **stability** are first-class citizens (not afterthought metrics).
5.  Time is an attribute, **not the organizing axis**. The organizing axis is **importance**.

2\. Node Taxonomy
-----------------

### 2.1 Core Node Types

#### `Person`

Represents a human (user, agent, prospect, owner, etc.).

Key fields:

*   `person_id` (PK)
*   `role` (e.g., owner, prospect, agent, internal\_user)
*   `traits` (JSON: stable attributes, e.g., risk\_tolerance, communication\_style)
*   `created_at`
*   `updated_at`

Derived metrics:

*   `trust_score` (global trust, 0–1)
*   `engagement_score`
*   `lifetime_value_estimate`

* * *

#### `InteractionSession`

A contiguous conversation context (call, meeting, multi-turn chat).

*   `session_id` (PK)
*   `channel` (phone, WhatsApp voice, Zoom, etc.)
*   `start_time`
*   `end_time`
*   `participants` (list of `person_id`)
*   `context_tags` (e.g., “onboarding”, “renewal”, “support\_escalation\`)

Aggregates:

*   `emotional_trajectory` (vector over time)
*   `conflict_index`
*   `resolution_probability`

* * *

#### `Utterance`

Atomic unit of human speech or agent output.

*   `utterance_id` (PK)
*   `session_id` (FK)
*   `speaker_id` (FK → Person or Agent)
*   `timestamp` (within session)
*   `transcript_text`
*   `language`
*   `nlp_intent` (string / enum)
*   `nlp_entities` (list)

Neuro-linguistic features (voice-derived):

*   `paralanguage` (JSON: pitch, cadence, volume, pauses, emphasis)
*   `emotion` (JSON: valence, arousal, discrete emotion scores)
*   `cognitive_load_score` (0–1)
*   `certainty_score` (0–1)
*   `commitment_score` (0–1)
*   `trust_delta` (relative change inferred from this utterance)
*   `importance_local` (how important this utterance is in its session, 0–1)

* * *

#### `Concept`

Abstract ideas / topics being discussed.

*   `concept_id` (PK)
*   `label` (e.g., “AGI”, “occupancy rate”, “pricing strategy”)
*   `type` (domain\_concept, product\_feature, KPI, belief, constraint)
*   `description`
*   `domain` (real\_estate, AI\_architecture, finance, etc.)

Derived:

*   `global_importance` (0–1 across graph)
*   `volatility_score` (how often beliefs about this concept change)

* * *

#### `State`

Represents a **latent state** of a Person or system at a point in graph evolution.

Examples:

*   “Owner is 70% likely to upgrade plan.”
*   “User is frustrated and ready to churn.”
*   “Prospect is 80% persuaded but financially constrained.”

Fields:

*   `state_id` (PK)
*   `owner_type` (Person / System / Relationship)
*   `owner_id`
*   `state_type` (e.g., `purchase_intent`, `satisfaction`, `risk_level`)
*   `value` (JSON: e.g., `{ "probability": 0.78 }`)
*   `confidence` (0–1)
*   `stability` (0–1; how quickly this state tends to change)
*   `last_updated_at`

* * *

#### `Outcome`

Actual realized event.

Examples:

*   “Booking extended.”
*   “Offer accepted.”
*   “Owner churned.”
*   “Tenant upgraded to premium.”

Fields:

*   `outcome_id` (PK)
*   `type` (conversion, churn, upgrade, NPS\_response, etc.)
*   `owner_type` (Person / Property / Account / Session)
*   `owner_id`
*   `value` (JSON: e.g., `{ "revenue_delta": 350.00 }`)
*   `timestamp`
*   `confidence` (0–1, for noisy or inferred outcomes)

* * *

#### `MemoryNode`

Causal memory abstraction – what the system stores **as a meaningful, reusable chunk**.

Typical MemoryNode examples:

*   “Todd distrusts Copilot-like agents due to repeated failure.”
*   “Jeremiah believes memory must be importance-based, not time-based.”
*   “This property owner shows high sensitivity to risk but high openness to data-driven optimization.”

Fields:

*   `memory_id` (PK)
*   `owner_type` (Person / System / Organization)
*   `owner_id`
*   `summary_text` (natural language summary)
*   `source_evidence` (list of `utterance_id` / `session_id` / `outcome_id`)
*   `importance_global` (0–1)
*   `importance_contextual` (per domain, per goal)
*   `trust_score` (0–1; how reliable this memory is)
*   `decay_rate` (0–1; how fast this memory decays if not reinforced)
*   `last_reinforced_at`
*   `created_at`

3\. Edge Taxonomy
-----------------

Edges encode **directional influence and relationships**.  
Core design principle: all causal reasoning flows along these edges.

### 3.1 Structural Edges (Context & Containment)

*   `UTTERED_IN`
    *   `(Utterance) → (InteractionSession)`
*   `PART_OF_SESSION`
    *   `(Utterance) → (InteractionSession)` (alias / synonym)
*   `PARTICIPATED_IN`
    *   `(Person) → (InteractionSession)`
*   `MENTIONS`
    *   `(Utterance) → (Concept)`

Fields:

*   `edge_id`
*   `edge_type`
*   `confidence`
*   `created_at`

* * *

### 3.2 Semantic & Attitudinal Edges

*   `EXPRESSES_OPINION_ON`  
    `(Utterance) → (Concept)`
    *   `polarity` (-1..1)
    *   `strength` (0–1)
    *   `emotion_profile` (JSON)
    *   `certainty` (0–1)
*   `HOLDS_BELIEF`  
    `(Person) → (Concept)`
    *   `belief_strength` (0–1)
    *   `direction` (support/oppose/neutral)
    *   `confidence`
    *   `last_updated_at`
*   `IN_STATE`  
    `(Person) → (State)`
    *   `start_time`
    *   `end_time` (optional)
    *   `derived_from` (list of `utterance_id`/`session_id`)

* * *

### 3.3 Causal Edges

These are the backbone.

*   `CAUSES`  
    `(Node A) → (Outcome or State B)`
    *   `causal_strength` (0–1)
    *   `lag_distribution` (time offset distribution)
    *   `evidence` (list of evidentiary node/edge IDs)
    *   `confidence`
*   `INFLUENCES`  
    `(Node A) → (Node B)`  
    Used when causality is suspected but not fully confirmed.
    *   `influence_strength`
    *   `direction` (positive/negative/mixed)
    *   `confidence`
*   `MODIFIES_STATE`  
    `(Utterance or InteractionSession) → (State)`
    *   `delta` (JSON: e.g., `{ "probability": +0.12 }`)
    *   `evidence_emotion` (from voice features)
    *   `evidence_cognition`
*   `REINFORCES_MEMORY`  
    `(Utterance or Outcome) → (MemoryNode)`
    *   `reinforcement_strength` (0–1)
    *   `last_reinforced_at`
    *   `impact_on_importance` (delta)
*   `CONTRADICTS_MEMORY`  
    `(Utterance or Outcome) → (MemoryNode)`
    *   `contradiction_strength`
    *   `impact_on_trust` (negative delta)
    *   `resolution_status` (open/resolved)

* * *

### 3.4 Importance & Trust Flow Edges

*   `DERIVES_IMPORTANCE_FROM`  
    `(MemoryNode) → (Concept/Outcome/Goal)`
    *   `weight` (how much this node matters for that goal)
*   `SUPPORTED_BY`  
    `(State or Belief or MemoryNode) → (Utterance/Outcome)`
    *   `evidence_strength`
    *   `evidence_type` (voice\_emotion, outcome\_data, behavioral\_pattern)

4\. Importance & Decay Logic (Core to the Schema)
-------------------------------------------------

These are not just business rules; they’re **schema-level behaviors**.

### 4.1 Node-Level

Every `MemoryNode`, `State`, `Belief`, and `Concept` carries:

*   `importance_global` (0–1)
*   `importance_contextual` (dict keyed by context/goal)
*   `decay_rate` (0–1)
*   `last_reinforced_at`

### 4.2 Update Rules (High-Level)

On each new interaction / utterance:

1.  **Compute local importance** (`importance_local`) from:
    *   emotion intensity
    *   cognitive load
    *   decision-state relevance
    *   proximity to active goals
2.  **Update related MemoryNodes**:
    *   If consistent with existing memory → increase `trust_score`, reduce decay, bump `importance_global`.
    *   If contradicting → decrease `trust_score`, maybe spawn alternate MemoryNode (forked belief).
3.  **Recalculate State probabilities**:
    *   Use `MODIFIES_STATE` edges from new utterance → update `State.value.probability`.
4.  **Reweight graph**:
    *   Run importance propagation:
        *   If a high-importance Concept is affected, propagate importance along `DERIVES_IMPORTANCE_FROM`, `CAUSES`, `INFLUENCES` edges.
